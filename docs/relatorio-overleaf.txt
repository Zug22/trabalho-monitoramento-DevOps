\begin{abstract}
Este relatório apresenta o desenvolvimento de uma plataforma de monitoramento para DevOps, construída para acompanhar serviços de rede e componentes de infraestrutura em ambiente conteinerizado. A solução monitora um servidor web desenvolvido em C++, um banco de dados PostgreSQL, serviços DNS e SMTP, além de eventos relacionados à segurança, como picos de tráfego, falhas de autenticação, alterações de configuração e vulnerabilidades conhecidas simuladas. A arquitetura utiliza Docker Compose, Prometheus, Grafana, Alertmanager, Gmail SMTP, Postgres Exporter e Blackbox Exporter. O resultado é um ambiente integrado para coleta de métricas, visualização em dashboards, emissão de alertas por e-mail e execução de testes controlados.

\vspace{0.3cm}
\noindent\textbf{Palavras-chave:} redes de computadores; monitoramento; DevOps; Prometheus; Grafana; segurança.
\end{abstract}

\section{Introdução}

Ambientes computacionais modernos dependem de diversos serviços operando de forma integrada. Servidores web, bancos de dados, serviços de resolução de nomes, canais de e-mail e mecanismos de autenticação precisam estar disponíveis e funcionando com desempenho adequado. Quando um desses componentes apresenta falha, lentidão ou comportamento anormal, a equipe responsável precisa identificar rapidamente a causa e responder ao incidente.

Nesse contexto, plataformas de monitoramento são fundamentais. Elas permitem acompanhar métricas de disponibilidade, latência, volume de acessos, taxa de erros, conexões ativas e eventos de segurança. Além disso, dashboards e alertas automatizados facilitam a observação contínua do ambiente e reduzem o tempo necessário para detectar e tratar problemas.

O projeto desenvolvido consiste em uma plataforma de monitoramento para DevOps, com foco em serviços de rede. A solução foi estruturada para atender aos requisitos propostos na disciplina de Redes de Computadores e Internet, contemplando front-end responsivo, dashboards, backend de métricas, banco de dados, regras de alerta, notificações por e-mail e documentação operacional.

\section{Objetivos}

\subsection{Objetivo Geral}

Projetar e implementar uma plataforma de monitoramento capaz de acompanhar a saúde de serviços de rede e infraestrutura, apresentando métricas em dashboards e notificando situações de degradação ou indisponibilidade.

\subsection{Objetivos Específicos}

\begin{itemize}
    \item Implementar um front-end responsivo com visão consolidada dos serviços monitorados.
    \item Desenvolver um backend em C++ para simular um serviço web e expor métricas em formato compatível com Prometheus.
    \item Adicionar um banco de dados PostgreSQL ao ambiente e monitorar suas métricas.
    \item Monitorar disponibilidade e tempo de resposta de serviços HTTP, DNS e SMTP.
    \item Configurar dashboards no Grafana para visualização das métricas.
    \item Criar regras de alerta com níveis de atenção e criticidade.
    \item Enviar notificações de alerta por e-mail por meio do Gmail SMTP.
    \item Implementar cenários de teste relacionados a segurança e comportamento anormal.
    \item Documentar arquitetura, execução, testes e resposta a incidentes.
\end{itemize}

\subsection{Fluxo de Dados}

O backend C++ expõe métricas no endpoint \texttt{/metrics}. O Prometheus coleta essas métricas periodicamente, juntamente com dados do Postgres Exporter e do Blackbox Exporter. O Grafana consulta o Prometheus para renderizar os dashboards. Quando uma condição de alerta é atendida, o Prometheus encaminha o alerta ao Alertmanager. O Alertmanager, por sua vez, envia a notificação por e-mail utilizando o SMTP do Gmail.

\section{Tecnologias Utilizadas}

\subsection{C++ e Crow}
O backend foi implementado em C++ com a biblioteca Crow. A aplicação simula um servidor web real, processa requisições, registra contadores e expõe métricas em texto no formato utilizado pelo Prometheus.

\subsection{Docker Compose}
O Docker Compose foi utilizado para orquestrar os serviços do ambiente. A plataforma inclui contêineres para backend, PostgreSQL, Postgres Exporter, Prometheus, Blackbox Exporter, Alertmanager e Grafana.

\subsection{Prometheus}
O Prometheus atua como mecanismo central de coleta e armazenamento de séries temporais. Ele executa scraping periódico dos endpoints configurados e avalia as regras de alerta.

\subsection{Grafana}
O Grafana é responsável pela visualização das métricas. O projeto utiliza provisionamento automático de datasource e dashboard, permitindo que os painéis estejam disponíveis assim que o ambiente é iniciado.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{dashboard-grafana.jpeg}
    \caption{Dashboard do Grafana com métricas do web server, banco PostgreSQL, probes de rede e eventos de segurança.}
    \label{fig:grafana-dashboard}
\end{figure}

\subsection{Alertmanager e Gmail SMTP}
O Alertmanager organiza e encaminha alertas. O envio de e-mail foi configurado por meio do servidor \texttt{smtp.gmail.com}, na porta 587, com TLS. Dessa forma, os alertas gerados pelo Prometheus são entregues em uma caixa de entrada real, permitindo validar o fluxo completo de notificação.

\subsection{PostgreSQL e Postgres Exporter}
O PostgreSQL representa o banco de dados da plataforma. O Postgres Exporter coleta métricas do banco e as disponibiliza para o Prometheus.

\subsection{Blackbox Exporter}
O Blackbox Exporter executa testes externos de disponibilidade e tempo de resposta. No projeto, ele é utilizado para monitorar HTTP, DNS e SMTP.

\section{Implementação do Backend}

O backend C++ disponibiliza tanto a interface visual quanto os endpoints de métricas e simulação. As principais rotas implementadas são:

\begin{itemize}
    \item \texttt{/}: dashboard principal da plataforma.
    \item \texttt{/ataques}: laboratório de ataques e simulações controladas.
    \item \texttt{/api/trabalho}: rota que simula a aplicação web monitorada.
    \item \texttt{/api/simular/login-falho}: simula falhas de autenticação.
    \item \texttt{/api/simular/configuracao}: simula alteração de configuração.
    \item \texttt{/api/simular/vulnerabilidade}: ativa uma vulnerabilidade conhecida simulada.
    \item \texttt{/api/simular/vulnerabilidade/resolver}: resolve a vulnerabilidade simulada.
    \item \texttt{/metrics}: expõe as métricas consumidas pelo Prometheus.
\end{itemize}

A rota \texttt{/api/trabalho} simula o comportamento de um serviço web. Ela incrementa contadores de requisições, registra conexões ativas, calcula latência e gera erros HTTP de forma controlada. Com isso, é possível observar variações nas métricas durante os testes.

\section{Front-end da Plataforma}

O projeto possui duas telas principais. A primeira é o painel de monitoramento em \texttt{http://localhost:8080}. Essa tela apresenta os indicadores principais do ambiente, incluindo disponibilidade, CPU, requisições, erros HTTP, latência, conexões ativas, falhas de login e vulnerabilidades.

A segunda tela é o laboratório de ataques, disponível em \texttt{http://localhost:8080/ataques}. Ela foi criada para facilitar a apresentação do trabalho e permitir a geração de eventos diretamente pelo navegador.

\subsection{Dashboard Principal}
O dashboard principal apresenta uma visão consolidada dos serviços monitorados. Ele também contém atalhos para Prometheus, Grafana, Gmail e laboratório de ataques. A interface foi construída de forma responsiva, permitindo uso em diferentes tamanhos de tela.

\section{Banco de Dados}

O banco de dados utilizado é o PostgreSQL. Ele é iniciado automaticamente pelo Docker Compose e recebe um script de inicialização com tabelas para métricas e incidentes. A inclusão do banco atende ao requisito de armazenamento da plataforma e permite monitorar um serviço adicional.

As principais informações monitoradas no banco incluem disponibilidade, número de conexões abertas, transações, rollbacks e tamanho do banco.

\section{Métricas Monitoradas}

\subsection{Web Server}
O servidor web C++ expõe as seguintes métricas:
\begin{itemize}
    \item \texttt{servidor\_web\_up}: indica se o servidor está disponível.
    \item \texttt{total\_requisicoes\_http}: total acumulado de requisições.
    \item \texttt{total\_erros\_http}: total acumulado de erros HTTP.
    \item \texttt{ultima\_latencia\_ms}: latência da última resposta em milissegundos.
    \item \texttt{conexoes\_ativas}: quantidade de conexões ativas no momento.
\end{itemize}

\subsection{Banco de Dados}
O monitoramento do PostgreSQL é feito pelo Postgres Exporter. As principais métricas utilizadas são:
\begin{itemize}
    \item \texttt{pg\_up}: disponibilidade do banco.
    \item \texttt{pg\_stat\_activity\_count}: número de conexões abertas.
    \item \texttt{pg\_stat\_database\_xact\_commit}: transações confirmadas.
    \item \texttt{pg\_stat\_database\_xact\_rollback}: transações revertidas.
    \item \texttt{pg\_database\_size\_bytes}: tamanho do banco.
\end{itemize}

\subsection{DNS}
O DNS é monitorado por meio do Blackbox Exporter. O probe verifica se uma consulta DNS pode ser resolvida corretamente e se o serviço responde dentro do tempo esperado.

\subsection{SMTP}
O SMTP é monitorado por um probe TCP direcionado ao endereço \texttt{smtp.gmail.com:587}. Dessa forma, é possível verificar se o canal externo utilizado para envio de e-mails está acessível a partir do ambiente.

\subsection{Segurança}
Os eventos de segurança monitorados são:
\begin{itemize}
    \item Pico de tráfego, utilizado como indicador de possível ataque de negação de serviço.
    \item Falhas de autenticação, associadas a tentativas de brute force.
    \item Alterações de configuração, detectadas por meio de checksum.
    \item Vulnerabilidade conhecida simulada, usada para demonstrar um alerta crítico.
\end{itemize}

\section{Notificações por E-mail}

O envio de notificações é feito pelo Alertmanager. No ambiente implementado, o Alertmanager utiliza o servidor SMTP do Gmail, com TLS na porta 587, para encaminhar os alertas gerados pelo Prometheus. Essa configuração permite demonstrar o envio real de e-mails a partir de uma condição crítica detectada pelo sistema de monitoramento.

O fluxo de notificação ocorre da seguinte forma:
\begin{enumerate}
    \item O Prometheus avalia uma regra de alerta.
    \item A condição permanece ativa pelo tempo configurado.
    \item O alerta é enviado ao Alertmanager.
    \item O Alertmanager encaminha o e-mail pelo Gmail SMTP.
    \item O e-mail pode ser visualizado na caixa de entrada do Gmail configurada no projeto.
\end{enumerate}

\section{Testes e Validação}

A validação da plataforma envolve navegação, consulta de métricas, geração de carga e acionamento de alertas. Os testes podem ser feitos pelo laboratório de ataques ou por linha de comando.

\subsection{Teste de Navegação}
\begin{enumerate}
    \item Abrir o front principal em \url{http://localhost:8080}.
    \item Verificar os cards de métricas.
    \item Abrir o laboratório em \url{http://localhost:8080/ataques}.
    \item Abrir o Grafana em \url{http://localhost:3000}.
    \item Consultar os targets do Prometheus em \url{http://localhost:9090/targets}.
    \item Verificar a caixa de entrada do Gmail após o disparo de alertas.
\end{enumerate}

\subsection{Teste de Carga HTTP}
O teste de carga HTTP pode ser executado pelo laboratório de ataques ou pelo comando:

\begin{lstlisting}[language=bash]
ab -n 20000 -c 100 http://localhost:8080/api/trabalho
\end{lstlisting}

Esse teste aumenta o volume de requisições e permite observar RPS, latência, conexões ativas e erros HTTP.

\subsection{Teste de Falhas de Autenticação}
\begin{lstlisting}[language=bash]
for i in {1..6}; do curl -i http://localhost:8080/api/simular/login-falho; done
\end{lstlisting}

Esse teste incrementa a métrica de falhas de login e permite acionar o alerta de autenticação.

\subsection{Teste de Vulnerabilidade Conhecida}
\begin{lstlisting}[language=bash]
curl http://localhost:8080/api/simular/vulnerabilidade
curl http://localhost:8080/api/simular/vulnerabilidade/resolver
\end{lstlisting}

O primeiro comando ativa a vulnerabilidade simulada, enquanto o segundo remove a condição crítica.

\subsection{Teste de Alteração de Configuração}
\begin{lstlisting}[language=bash]
curl http://localhost:8080/api/simular/configuracao
\end{lstlisting}

Esse teste altera uma métrica de checksum simulada e permite demonstrar o alerta de alteração de configuração.

\section{Playbooks de Incidentes}

\subsection{CPU Alta}
Quando houver alerta de CPU alta, deve-se verificar se há teste de carga em andamento, consultar os gráficos do Grafana e analisar os logs do backend.

\subsection{Erros HTTP}
Em caso de aumento de erros HTTP, deve-se verificar a rota \texttt{/api/trabalho}, observar a taxa de erro no Grafana e confirmar se há simulação ativa no laboratório de ataques.

\subsection{Banco Indisponível}
Para alerta de banco indisponível, deve-se verificar o container do PostgreSQL, consultar os logs do banco e confirmar se o Postgres Exporter está respondendo.

\subsection{Falhas de Autenticação}
Quando houver muitas falhas de autenticação, deve-se verificar se a simulação de brute force foi executada. Em um ambiente real, a resposta envolveria bloqueio temporário, análise de origem e reforço de autenticação.

\subsection{Vulnerabilidade Conhecida}
Quando a vulnerabilidade simulada estiver ativa, o alerta crítico demonstra a necessidade de identificar o serviço afetado, corrigir a versão vulnerável e validar a resolução.

\section{Resultados}

O projeto resultou em uma plataforma funcional de monitoramento. A solução permite observar métricas do web server, banco de dados, DNS, SMTP e eventos de segurança. Também permite gerar cenários de teste diretamente pelo navegador, facilitando a demonstração dos alertas.

Os principais resultados alcançados foram:
\begin{itemize}
    \item Dashboard próprio responsivo.
    \item Laboratório de ataques controlados.
    \item Dashboard Grafana provisionado automaticamente.
    \item Coleta de métricas pelo Prometheus.
    \item Monitoramento de banco PostgreSQL.
    \item Probes HTTP, DNS e SMTP.
    \item Envio de alertas reais por e-mail usando Gmail SMTP.
    \item Documentação técnica e operacional.
\end{itemize}

\section{Limitações}

Algumas métricas avançadas foram simplificadas para adequação ao ambiente acadêmico. Por exemplo, slow queries reais, autenticação com usuários reais e persistência automática de incidentes no banco não foram implementadas de forma produtiva. No entanto, o projeto apresenta simulações controladas suficientes para demonstrar o comportamento da plataforma e o acionamento dos alertas.

\section{Trabalhos Futuros}

Como trabalhos futuros, seria possível:
\begin{itemize}
    \item Implementar autenticação real no front-end.
    \item Persistir incidentes automaticamente no PostgreSQL.
    \item Coletar logs estruturados.
    \item Adicionar métricas reais de slow queries.
    \item Implantar a plataforma em ambiente de nuvem.
    \item Adicionar autenticação e controle de acesso aos dashboards.
    \item Substituir credenciais fixas por gerenciamento externo de segredos.
\end{itemize}

\section{Conclusão}

A plataforma desenvolvida demonstra como ferramentas de observabilidade podem ser integradas para monitorar serviços de rede e aplicações. Com Prometheus, Grafana, Alertmanager, Gmail SMTP, PostgreSQL e exporters, foi possível construir um ambiente completo para coleta de métricas, visualização, alertas e testes controlados.

O projeto atende aos principais requisitos propostos no trabalho final, incluindo front-end responsivo, dashboards por serviço, backend de métricas, banco de dados, alertas por e-mail, documentação e monitoramentos extras de segurança. A solução também apresenta um laboratório de ataques controlados, que facilita a validação dos alertas e a demonstração prática da plataforma.
